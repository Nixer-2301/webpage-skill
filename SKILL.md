# Web Creator – 网页生成工具

## 核心原则
1. **内容不变**：无论用户是新手还是专家，生成的最终视觉效果、布局、文本内容完全一致。
2. **注释按需**：仅代码注释的详细程度和额外说明根据用户身份调整（资深大牛 → 简洁专业注释；入门小白 → 详尽解释、分步说明）。
3. **版本迭代**：用户每次提出修改建议，生成新的版本文件夹（小改升级小版本如1.0→1.1；大改升级大版本如1.0→2.0）。

## 工作流程（严格执行）

### 步骤0：识别用户并通过合适的方式识别及询问用户的需求
分析用户描述及对话历史，判断其身份，若未说明具体需求，请询问用户：
```
您好！欢迎使用Webpage-skill！本skill用于创建网页，请输入您的需求，例如需要什么样式的界面，会带来什么感觉等。
```
- **判定依据**：
  - 使用专业术语（如“响应式”、“BEM命名”、“Tree Shaking”）、提及框架/构建工具 → **资深大牛**
  - 描述笼统（如“好看的按钮”、“放中间”）、询问基础问题 → **入门小白**
  - 无法判断 → 默认 **入门小白**（确保注释足够详细）
- **影响范围**：
  - 代码中的注释风格（详见步骤5）。
  - 最终交付时的额外口头说明（对小白可补充“如何修改颜色/文字”等提示）。

### 步骤1：确保工作区存在web目录
- 检查当前工作区根目录下是否存在 `web` 文件夹。
- 若不存在，执行：`mkdir web`（或使用Bash工具创建）。
- 若已存在，直接进入下一步。

### 步骤2：创建项目文件夹（命名和版本结构）
1. **提取项目名称**：根据用户对网页的描述，总结为**短横线小写**的英文/数字组合（如 `personal-dashboard`、`product-landing`）。
2. **创建项目目录**：在 `web/` 下创建 `{项目名}` 文件夹（例：`web/personal-dashboard`）。
3. **确定版本号**：首次创建使用 `v1.0`。后续迭代时根据用户修改幅度决定：
   - 小改动（文案微调、颜色微调、间距调整） → 只增加补丁版本或小版本：`v1.0` → `v1.1`
   - 大改动（新增区块、改变布局、增加交互功能） → 升级大版本：`v1.0` → `v2.0`
4. **创建版本文件夹**：在项目目录下创建 `{版本号}` 文件夹（例：`web/personal-dashboard/v1.0`）。

### 步骤3：分析需求，列出所需文件清单
根据用户诉求，确定要生成的文件。通常包括：
- `index.html`（必须）
- `style.css` 或 `main.css`（推荐独立，除非极小项目）
- `script.js`（如有交互）
- 字体文件（若使用自定义字体，放在 `fonts/` 或使用CDN）
- 图片资源（若需要，可生成占位图或引用unsplash等）
可以参考skill文件内的资源文件夹内的不同案例，因人而异，自行判断。


### 步骤4：构思排版与设计（确保一致性、美观性）
- 参考本技能内置的 `references/design-patterns.md`（若存在）或常见优秀网页布局：
  - 首屏清晰（Hero + 标题/按钮）
  - 内容区网格/卡片
  - 响应式（移动端适配）
  - 配色和谐（主色+辅色+中性色）
- 保持页面**整体风格统一**（圆角、阴影、间距等一致）。
- 若用户要求“壮观”、“高级感”，按之前 `webcreator` 思路融入动态背景或渐变。

**重要**：无论用户水平如何，最终渲染的HTML/CSS/JS视觉效果完全相同。仅注释区别。

### 步骤5：生成第一版文件（内容与注释）
**文件生成规则**：
1. 所有文件必须保存在步骤2确定的版本文件夹内（如 `web/项目名/v1.0/`）。
2. **HTML结构**：语义化标签，必要meta视口设置，外部链接CSS/JS。
3. **CSS布局**：优先Flex/Grid，移动优先媒体查询。
4. **JS交互**：如有，使用现代ES6语法并附必要注释。
5. **注释规则**（根据步骤0识别的用户身份）：
   - **资深大牛**：注释简明扼要，只点明关键逻辑、复杂计算、浏览器兼容性hack。不解释基础语法。
     ```css
     /* Hero section – full viewport with gradient overlay */
     ```
   - **入门小白**：注释详细，解释每个区块的作用、属性含义、如何修改常见项（颜色、间距）。
     ```css
     /* Hero区域 – 占满整个屏幕高度，背景是一个深色渐变
        若要修改背景色：更改下面linear-gradient中的颜色值 */
     ```

**特别约束**：保证视觉内容完全相同。例如，即使是小白版本，也不能为了解释而多添加一个不影响布局的元素。

### 步骤6：交付第一版并提示用户
生成所有文件后，回复用户以下消息（必须包含具体路径和版本）：
```

✅ 您的第一版网页文件生成完成！

📁 项目位置：[工作区绝对路径]/web/{项目名}/{版本号}/
🌐 打开方式：用浏览器打开该文件夹中的 index.html 文件

💡 如需修改：

· 请直接告诉我改进建议（如“把按钮改成红色”、“增加一个图片轮播”）
· 我会自动创建新版本文件夹（如从 v1.0 升级到 v1.1 或 v2.0），不影响旧版本。

感谢使用网页生成器！

```

### 步骤7：迭代改进（当用户提出新建议时）
1. **分析修改幅度**：
   - 小改（文案、颜色、间距、单个样式调整） → 新版本号 = 当前版本号小版本+1（如 v1.0 → v1.1）
   - 大改（新增区块、改变布局结构、增加复杂JS功能） → 新版本号 = 大版本+1（如 v1.0 → v2.0）
2. **创建新版本文件夹**：在项目目录下新建 `{新版本号}` 文件夹（保留旧版本）。
3. **重复步骤3-6**：基于用户新的描述，生成改进后的完整文件集（注意：需要重新思考所需文件，但通常与上一版类似，可复制修改）。
4. **交付时注明版本变化**：

```
✅ 已根据您的建议生成改进版本 v1.1。
修改内容：按钮颜色由蓝色改为红色，全局字体间距增大。
旧版本保留在 v1.0 文件夹中。
```

## 额外约束
- **禁止改变视觉内容**：无论用户身份，最终HTML渲染的页面内容、布局、文字、图片位置必须完全一致。
- **宁可2000行注释，不多加一个div**：注释详细程度可以区分，但禁止为了“解释”而增加额外元素。
- **版本管理清洁**：每次迭代生成新版本文件夹，不覆盖旧版本。
- **路径安全**：所有文件写入必须位于工作区的 `web/` 下，不得越权。

## 错误处理
- **目录已存在但版本号冲突**：如果用户要求生成的项目名+版本号已经存在，询问用户是否覆盖或升级版本。
- **用户未描述足够信息**：主动提问（至少问“页面的主题是什么？需要展示什么内容？”），但一次最多提2个问题，避免让小白感到压力。

## 快速示例（供AI内部参考）
### 小白用户输入：
> “帮我做一个个人介绍页面，要有姓名、照片和兴趣爱好。”

**AI内部行动**：
- 身份判定：小白 → 注释详尽，代码后附解释。
- 生成 `web/personal-intro/v1.0/` 下 `index.html`, `css/style.css`, `js/script.js`（可能无交互）。
- HTML中包含头像占位图（或Font Awesome图标）。
- CSS注释解释每个区块的用途，并写“修改背景颜色请改这里”。

### 大牛用户输入：
> “生成一个响应式作品集网格，支持暗色模式，使用CSS Grid和flex混合布局。”

**AI内部行动**：
- 身份判定：大牛 → 注释只写为什么用Grid、暗色模式的prefers-color-scheme兼容处理。
- 生成完全相同的视觉作品集（与小白看到的一模一样）。
- 代码中不解释基础概念，但会有关键点注释。

## 附录：常用设计参考（可选）
当需要生成“壮观”或“高级感”首屏时，可参考以下模式：
- 全屏渐变 + 动态光晕
- 粒子背景（轻量Canvas）
- 大号可变字体 + 超大留白
- 按钮悬浮流光边缘
- 滚动触发的淡入动画（Intersection Observer）

以上所有模式必须保证响应式，并在移动端可读。
